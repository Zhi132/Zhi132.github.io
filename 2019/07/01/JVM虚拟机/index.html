前言

本篇博客是关于JVM虚拟机一些见解

😊JVM虚拟机

一、JVM结构

1、server端与client端的区别与使用。

  • Java HotSpot Client VM：为在客户端环境中减少启动时间而优化
  • Java HotSpot Server VM：为在服务器环境中最大化程序执行速度而设计
  • 区别
    • cient使用的是一个代号为C1的轻量级编译器
    • server模式启动的虚拟机采用相对重量级，代号为C2的编译器
    • C2比C1编译器编译的相对彻底，服务起来之后，性能更高。

2、基本结构

2.1 类加载子系统与方法区
  • 通过类加载子系统从文件系统或者网络中加载class信息，并将其放入方法区中，以及类中的常量等。
  • 方法区一般放的都是唯一的。
2.2 Java堆
  • 在虚拟机启动的时候建立
  • 主要内存工作区域
  • 几乎所有的java对象实例都放在java堆中
  • 线程共享的
2.3 直接内存
  • NIO库使用
  • 大小不会受限于xmx
2.4 垃圾回收系统
  • 对象空间的释放是透明的
  • 可以对方法区和java堆和直接内存进行回收
2.5 Java栈
  • 在线程创建的时候被创建
  • 保存帧信息。保存局部变量、方法参数、同时和java方法的调用、返回密切相关。
2.6 本地方法栈(nativemethod)
  • 本地方法的调用-----离不开操作系统
  • 通常使用c编写
2.7 PC寄存器
  • 线程私有的空间
  • 区分线程执行的方法时本地还是java自己的方法
2.8 执行引擎
  • 最核心的组件之一
  • 即时编辑(just in time)

3、JVM堆结构图及分代

3.1 JVM分代策略
  • Java虚拟机根据对象存活的周期不同，把堆内存划分为几块，一般分为新生代、老年代和永久代（对HotSpot虚拟机而言），这就是JVM的内存分代策略。
3.2 为什么要分代 ？
  • 提高对象内存分配和垃圾回收的效率
    • 产生碎片会造成浪费------如果不分代----如果想要用上就得把这一块不上（感觉和操作系统的最小分配机制可以联系起来）。
  • 没有内存分代需要多次遍历—效率低
  • 各分代回收
    • 新生代：新创建的对象会在新生代中分配内存，经过多次回收仍然存活下来的对象存放在老年代中
    • 静态属性、类信息等存放在永久代中
    • 新生代中的对象存活时间短，只需要在新生代区域中频繁进行GC
    • 老年代中对象生命周期长，内存回收的频率相对较低，不需要频繁进行回收
    • 永久代中回收效果太差，一般不进行垃圾回收。
  • 可以对不同的年代的特点使用合适的垃圾收集算法-----提高效率
