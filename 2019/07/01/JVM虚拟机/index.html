JVM虚拟机

发表于 2019-07-01 | 分类于 Java知识点 | 次阅读

前言

本篇博客是关于JVM虚拟机一些见解

😊JVM虚拟机

一、JVM结构

1、server端与client端的区别与使用。

  • Java HotSpot Client VM：为在客户端环境中减少启动时间而优化
  • Java HotSpot Server VM：为在服务器环境中最大化程序执行速度而设计
  • 区别
    • cient使用的是一个代号为C1的轻量级编译器
    • server模式启动的虚拟机采用相对重量级，代号为C2的编译器
    • C2比C1编译器编译的相对彻底，服务起来之后，性能更高。

2、基本结构

2.1 类加载子系统与方法区
  • 通过类加载子系统从文件系统或者网络中加载class信息，并将其放入方法区中，以及类中的常量等。
  • 方法区一般放的都是唯一的。
2.2 Java堆
  • 在虚拟机启动的时候建立
  • 主要内存工作区域
  • 几乎所有的java对象实例都放在java堆中
  • 线程共享的
2.3 直接内存
  • NIO库使用
  • 大小不会受限于xmx
2.4 垃圾回收系统
  • 对象空间的释放是透明的
  • 可以对方法区和java堆和直接内存进行回收
2.5 Java栈
  • 在线程创建的时候被创建
  • 保存帧信息。保存局部变量、方法参数、同时和java方法的调用、返回密切相关。
2.6 本地方法栈(nativemethod)
  • 本地方法的调用-----离不开操作系统
  • 通常使用c编写
2.7 PC寄存器
  • 线程私有的空间
  • 区分线程执行的方法时本地还是java自己的方法
2.8 执行引擎
  • 最核心的组件之一
  • 即时编辑(just in time)

3、JVM堆结构图及分代

3.1 JVM分代策略
  • Java虚拟机根据对象存活的周期不同，把堆内存划分为几块，一般分为新生代、老年代和永久代（对HotSpot虚拟机而言），这就是JVM的内存分代策略。
3.2 为什么要分代 ？
  • 提高对象内存分配和垃圾回收的效率
    • 产生碎片会造成浪费------如果不分代----如果想要用上就得把这一块不上（感觉和操作系统的最小分配机制可以联系起来）。
  • 没有内存分代需要多次遍历—效率低
  • 各分代回收
    • 新生代：新创建的对象会在新生代中分配内存，经过多次回收仍然存活下来的对象存放在老年代中
    • 静态属性、类信息等存放在永久代中
    • 新生代中的对象存活时间短，只需要在新生代区域中频繁进行GC
    • 老年代中对象生命周期长，内存回收的频率相对较低，不需要频繁进行回收
    • 永久代中回收效果太差，一般不进行垃圾回收。
  • 可以对不同的年代的特点使用合适的垃圾收集算法-----提高效率
3.3 讲解

3.3.1新生代
  • 回收效率很高（70-95）
  • 新生代为3块 比例-》8:1:1
    • 目的：hotspot使用复制算法来回收新生代
    • 新生成的对象在Eden区进行分配（大对象除外，直接进入老年代）
  • 对新生成的对象进行分配
    • 会发起一次MinorGc
    • GC开始时，对象只会存在于Eden去和From Survivor区
    • GC进行时，Eden去所有的存活对象都会被复制到To Survivor
      • 在From Survivor的会个悲剧年龄阈值来进行划分（默认为15），达到的划分到老年区，没有到达的划分到To Survivor。
3.3.2老年代
  • 回收速度慢
  • 存活比率高
3.3.3永久代
  • 永久存储类信息、常量、静态变量、即时编译器编译后的代码等数据。

4、JVM的垃圾回收算法

4.1 引用计数算法
  • 只回收收集计数为0的对象
  • 无法处理循环引用的问题
4.2 复制算法
  • 把正在使用的对象复制到另外一个区域中
  • 复制成本小，并且会进行相应的内存处理
  • 需要两倍内存空间
4.3 标记-清除算法------前辈
  • 第一阶段：从引用根节点开始标记所有被引用的对象。
  • 第二阶段：遍历整个堆，把未标记的对象清除。
  • 需要暂停整个应用（后续来解决并优化）-----同时，会产生碎片。
4.4 标记-整理算法
  • 结合了标记-清除和复制两个算法的优点
  • 第一阶段：从引用根节点开始标记所有被引用的对象。
  • 第二阶段：遍历整个堆，把未标记的对象清除。并且把存活的对象压缩到堆中的其中一块（有序）。

5、JVM中的垃圾收集器

​ 如果说垃圾收集算法是JVM内存回收的方法论，那么垃圾收集器就是内存回收的具体实现。

5.1 Scavenge GC(次收集)和Full GC(全收集)
  • Scavenge GC会很频繁
    • 用在年轻区
  • Full GC
    • 发生在老年代的GC，出现了Full GC一般会伴随着至少一次的Minor GC
    • 在无法回收年轻代的时候触发
5.2 次收集与全收集
  • 次收集
    • 当年轻代对空间紧张时会被触发
    • 相对于全收集而言，收集间隔较短
  • 全收集
    • 当老年代或者持久代堆空间满了，会触发全收集操作
    • 可以使用System.gc()方法来显示的启动全收集
    • 全收集一般会根据堆大小的不同，需要的时间不尽相同。
5.3 分代垃圾回收器
5.3.1新生代收集器

  • 5.3.1.1串行收集器（Serial）—单线程收集

    • 只用一个CPU/一条收集线程去完成GC工作
    • 必须暂停其他所有的工作线程
    • Serial收集器依然是虚拟机运行在Client模式下默认新生代收集器，对于运行在Client模式下的虚拟机来说是一个很好的选择。

  • 5.3.1.2 并行收集器（ParNew）-----多核心情况

    • 是Serial的多线程版本
    • 避免不了STOP THE WORLD
    • 是VM启用CMS收集器的默认新生代收集器
    • 在CPU数量过大的时候的环境中，可用-XX:ParalleIGCThreads=<数量>参数来控制GC线程数

  • 5.3.1.3 Parallel Scanvenge收集器

    • 并行多线程收集器，使用复制算法

    • 更加关注系统吞吐量，而不是安全点时间的问题

      系统吞吐量=运行用户代码时间/(运行用户代码时间+垃圾收集时间)

    • 停顿时间越短适用于用户交互的程序，提升用户的体验

    • 高吞吐量则适用于后台运算而不需要太多交互的任务-可以最高效率的利用CPU时间

    老年代收集器

  • 5.3.1.4 Serial Old收集器

    • 单线程收集器
    • 使用标记-整理算法
    • 搭配使用
      • 可以跟年轻代任何收集器相互使用，相互合作。

  • 5.3.1.5 Parallel Old收集器

    • 多线程收集器
    • 使用标记-整理算法
    • 搭配使用
      • parallel scavenge使用

  • 5.3.1.6 CMS收集器

    ​ 并行：CPU在停止用户程序的时候，GC是可以以多线程运行的

    ​ 并发：垃圾回收线程与用户工作线程交替运行

    fx：

    ​ 咖啡店两个队伍与一个队伍买咖啡

    并发：一个队伍

    并行：两个队伍 交互买咖啡

    • 以获取最短回收停顿时间为目标的收集器（又叫：多并发低暂停的收集器）

    • 基于“标记-清楚算法实现”

    • GC过程

      • 初始标记
      • 并发标记
      • 重新标记
      • 并发标记

      加粗位置需要进行STW

      初始位置只标记一下GC Roots能直接关联的对象

      重新标记是为了修正并发标记期间因用户程序继续运行而导致标记产生变动的那一部分对象的标记记录

    • 三个明显的缺点

      • CMS收集器无法处理浮动垃圾，可能出现“Concurrent Mode Failure”失败而导致另一次Full GC的产生。
      • 在JDK1.5的默认设置下，CMS收集器当老年代使用了68%的空间后就会被激活；
      • CMS是基于“标记-清除”算法实现的收集器，手机结束时会有大量空间碎片产生。空间碎片过多，可能会出现老年代还有很大空间剩余，但是无法找到足够大的连续空间来分配当前对象，不得不提前出发FullGC。

  • 5.3.1.7 分区收集-G1收集器（标记整理算法）

    • 面向服务端应用的收集器，主要用于配置多频CPU的服务器治理大内存
    • 优势
      • 独特的分代垃圾回收器,分代GC: 分代收集器, 同时兼顾年轻代和老年代；
      • 使用分区算法, 不要求eden, 年轻代或老年代的空间都连续；
      • 并行性: 回收期间, 可由多个线程同时工作, 有效利用多核cpu资源；
      • 空间整理: 回收过程中, 会进行适当对象移动, 减少空间碎片；
      • 可预见性: G1可选取部分区域进行回收, 可以缩小回收范围, 减少全局停顿。
    • 步骤
      • 1、初始标记（它标记了从GC Root开始直接可达的对象）；
      • 2、并发标记（从GC Roots开始对堆中对象进行可达性分析，找出存活对象）；
      • 3、最终标记（标记那些在并发标记阶段发生变化的对象，将被回收）；
      • 4、筛选回收（首先对各个Regin的回收价值和成本进行排序，根据用户所期待的GC停顿时间指定回收计划，回收一部分Region）。

9、一些问题总结与提出（参考https://www.cnblogs.com/xingzc/p/5756119.html）

9.1堆和栈的概括
  • 栈是运行时的单位 , 而堆是存储的单元。
  • 栈解决程序的运行问题，即程序如何执行，或者说如何处理数据，堆解决的是数据存储的问题，即数据怎么放，放在哪儿。
9.2为什么要把堆和栈区分出来？
  1. 从软件设计的角度看，栈代表了处理逻辑，而堆代表了数据。这样分开，使得处理逻辑更为清晰。分而治之的思想。这种隔离、模块化的思想在软件设计的方方面面都有体现。
  2. 堆与栈的分离，使得堆中的内容可以被多个栈共享（也可以理解为多个线程访问同一个对象）。

​ 好处:

​ a.提供了一种有效的数据交互方式（如：共享内存）

​ b.堆中的共享常量和缓存可以被所有栈访问，节省了空间。

  1. 栈因为运行时的需要，比如保存系统运行的上下文，需要进行地址段的划分。

​ 由于栈只能向上增长，因此就会限制栈存储内容的能力，

​ 而堆不同，堆中的对象是可以根据需要动态增长的，

​ 因此栈和堆的拆分使得动态增长成为可能，相应栈中只需记录堆中的一个地址即可。

  1. 面向对象就是堆和栈的完美结合。

​ 其实，面向对象方式的程序与以前结构化的程序在执行上没有任何区别。

​ 但是，面向对象的引入，使得对待问题的思考方式发生了改变，而更接近于自然方式的思考。

​ 当我们把对象拆开，你会发现，对象的属性其实就是数据，存放在堆中；

​ 而对象的行为（方法），就是运行逻辑，放在栈中。

​ 我们在编写对象的时候，其实就是编写了数据结构，也编写了处理数据的逻辑。不得不承认，面向对象的设计，确实很美。
